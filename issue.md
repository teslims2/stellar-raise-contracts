201. Design Strategy: Atomic Campaign State Transitions
Issue Title: Implement Robust State Management for Campaign Lifecycle
Description:
The campaign must move through four distinct states: Active, Successful, Failed, and Finalized. We need a design that prevents "state bleeding"—for example, ensuring a creator cannot withdraw funds while a campaign is still active, even if the goal has been met.
Proposed Requirements:
• Define a Status enum: Pending, Active, Succeeded, Expired.
• Implement logic to calculate status dynamically based on block.timestamp and totalPledged.
• Security: Ensure the Withdraw function explicitly requires the state to be Succeeded and the Refund function requires Expired.

202. Design Strategy: Pull-over-Push Refund Mechanism
Issue Title: Secure Refund Pattern to Prevent Gas Exhaustion
Description:
While the prompt mentions "automatic" refunds, a true "push-based" refund (sending tokens back to everyone at once) is a major security risk in blockchain design. It can lead to Denial of Service (DoS) if the contributor list is too long for the gas limit.
Proposed Requirements:
• Design a Pull-based architecture where the contract tracks individual mapping(address => uint256) pledges.
• Create a claimRefund() function that allows users to withdraw their specific balance.
• Update the Contribute logic to ensure funds are held in escrow by the contract, not sent to the creator immediately.

203. Design Strategy: Flexible Token Support & Validation
Issue Title: Standardized Asset Interface and Decimals Handling
Description:
To ensure "anyone" can create a campaign, the system must handle various ERC-20 (or equivalent) tokens. We need a design that handles different decimal counts (e.g., USDT’s 6 decimals vs. DAI’s 18) to avoid "dust" remaining in the contract or failed goals due to rounding errors.
Proposed Requirements:
• Implement a tokenAddress variable per campaign.
• Create a validation check during Initialize to ensure the token is a verified contract.
• Math: Use a consistent precision library to calculate if the goal has been met relative to the token's decimals.

204. Design Strategy: Campaign Discovery & Metadata Schema
Issue Title: Off-Chain Indexing and Event Emission Strategy
Description:
On-chain data is difficult to query for a UI. To make Stellar Raise "browsable," we need a standardized way to emit events so that indexers (like The Graph) can display active campaigns to potential contributors.
Proposed Requirements:
• Define Event structures for CampaignCreated, PledgeReceived, and FundsWithdrawn.
• Include a metadataURI (IPFS hash) in the Initialize function so creators can link to campaign descriptions, images, and social proof without bloating on-chain storage.
• Ensure the PledgeReceived event includes the total amount raised to date for real-time progress bars in the UI.

205. Design Strategy: Role-Based Access Control
Issue Title: Restrict Administrative Functions with RBAC
Description:
Campaign creators should not have absolute authority over system-level parameters (like platform fees or paused states). We need a clear separation of privileges to limit the blast radius if an admin key is compromised.
Proposed Requirements:
• Define distinct roles: DEFAULT_ADMIN_ROLE, CAMPAIGN_CREATOR, and PAUSER.
• Allow only authorized admins to pause the contract in the event of an emergency (Pausable logic).
• Require that setting platform fees is governed strictly by a multi-sig or DAO governance address rather than a single EOA.

206. Design Strategy: Reentrancy Attack Prevention
Issue Title: Implement Mutex Locks on Withdraw and Refund Functions
Description:
Since funds are held in escrow, any function that transfers tokens out of the contract is vulnerable to reentrancy attacks if it uses external calls. We must ensure state variables are updated before any external transfers occur.
Proposed Requirements:
• Implement a ReentrancyGuard utility or nonReentrant modifier for all state-changing token transfer functions.
• Follow the Checks-Effects-Interactions pattern rigidly within the `withdraw()` and `claimRefund()` functions.
• Ensure balances and status flags are zeroed/updated before transferring tokens out of the contract.

207. Design Strategy: Gas-Efficient Batch Processing
Issue Title: Optimize Multi-Campaign Interactions
Description:
As the platform scales, users might want to fund multiple campaigns in a single transaction or indexers might need to read multiple states. We need to reduce gas costs associated with iterative operations.
Proposed Requirements:
• Provide a `batchContribute()` function for users to support multiple projects simultaneously.
• Avoid using loops over unbound arrays (e.g., iterating over all campaigns) anywhere in on-chain logic.
• Use mappings extensively instead of arrays for storing core relationships (e.g. `mapping(uint256 => Campaign)`).

208. Design Strategy: Upgradeability and Storage Collisions
Issue Title: Ensure Safe Contract Upgrades via Proxy Pattern
Description:
To future-proof Stellar Raise, the smart contracts may need updates without migrating users to a new contract address. Poorly designed upgrade paths risk storage collisions and lost user funds.
Proposed Requirements:
• Implement an established Proxy pattern (e.g., UUPS or Transparent Proxy).
• Restrict the `upgradeTo()` function exclusively to the DAO or designated multi-sig admin.
• Avoid using `selfdestruct` or modifying the order of existing state variables to prevent storage slot corruption during upgrades.
